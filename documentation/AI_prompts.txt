Non-Functional Requirements
* Elasticity: Dynamic scale-up/down capabilities for processing components.

Option 3: Append-Only File Storage

Messages appended to file

**Implementation Plan for Option 3:**

**Phase 1 (MVP - Current Project):**
- 3 partitions with consistent hashing
- WAL per partition
- Basic metrics

**Comparison Across Criteria:**

| Criteria | Single Queue | Partitioned Queues |
|----------|-------------|-------------------|
| **Fault Tolerance** | ❌ Single point of failure | ✅ Partial availability (one partition down ≠ system down) |
| **Scalability** | ❌ Limited to single instance | ✅ Horizontal scaling by adding partitions |
| **Availability** | ❌ Queue down = system down | ✅ Some partitions up = partial service |
| **Scope for Improvement** | ⚠️ Requires redesign to scale | ✅ Can add replication, rebalancing, etc. |
| **Minimal Arch Change to Production** | ❌ Needs complete redesign for HA | ✅ Add replication per partition (same arch) |
| **No Data Loss** | ✅ Append-only file ensures durability | ✅ Append-only file per partition ensures durability |
| **Multiple Producer Instances** | ✅ Supported (concurrent writes) | ✅ Supported (write to specific partition) |
| **Multiple Consumer Instances** | ✅ Supported (consumer groups) | ✅ Supported (consumer groups per partition) |

**Winner: Partitioned Queues**
